# RobustQuant - 项目规范

## 核心原则

**先设计，后编码；先安全，后收益。** 任何交易、风控、券商接入、账户、订单、数据模型相关开发开始前，必须先与用户共同完成设计文档或更新现有设计，经用户确认后再进入代码实现。

RobustQuant 是量化交易项目，实盘相关改动可能影响真实资金。所有自动交易能力都必须可审计、可暂停、可撤单、可复盘。

## 工作流程

1. 与用户讨论并更新项目设计文档。
2. 明确需求边界、接口、数据模型、风控规则和验收标准。
3. 用户确认设计方向无误。
4. 方可进入编码阶段。

当前架构草案位于 `docs/01-architecture.md`，术语解释位于 `docs/02-glossary.md`，待决策问题位于 `docs/03-open-decisions.md`。后续若出现范围、接口、状态机、数据模型、风控规则、券商适配器行为或部署方式变更，必须先更新对应文档，再进入代码修改。

## 技术栈（暂定）

- 主要语言：Python。
- 后端 API：优先考虑 FastAPI。
- 数据库：本地开发可用 SQLite 起步，实盘前优先 PostgreSQL。
- 缓存 / 任务协调：可评估 Redis。
- 任务调度：可评估 APScheduler、Celery 或 Dramatiq。
- 前端：待确认，可按用户熟悉度和项目复杂度选择 React 或 Vue。
- 券商接入：miniQMT 用于东北证券等 QMT 账户；盈立证券 OpenAPI 用于港股、美股账户。
- AI / LLM：先作为研究辅助，不直接进入实盘交易链路。

技术栈尚未最终确认时，不要过早搭建复杂工程骨架。优先完成最小交易闭环的设计：账户查询、持仓查询、下单、撤单、订单查询、成交查询、风控、日志、对账。

## 交付目标

- 形成清晰、可迭代的量化交易系统设计文档。
- 建立高内聚、低耦合的核心架构：数据、策略、回测、风控、OMS、券商适配器、监控界面相互解耦。
- 第一阶段优先跑通安全的最小可用交易闭环，而不是追求复杂 AI 策略。
- 所有实盘行为都有日志、状态、错误原因和人工干预入口。

## Codex 工作约束

- 任务开始前先读 `docs/README.md` 和相关设计文档，确认当前真相源。
- 涉及架构、交易链路、风控、账户、订单、数据模型、部署方式的改动，先更新 `docs/`，再编码。
- 手工编辑文件使用 `apply_patch`；不得回滚用户已有改动。
- 代码改动后运行相关格式化、lint、typecheck、test；无法运行时说明原因。
- UI 改动必须做实际渲染或截图验证。
- 工具失败后先分析原因再换方式处理，禁止无脑重复同一失败调用。
- 不提交、不打印、不写入文档或代码中的券商 API Key、Secret、token、密码、真实账户号、真实身份证明、真实资金隐私数据。
- 不执行破坏性命令，除非用户明确要求并确认风险。
- 任务结束时汇报改动摘要、验证结果和遗留问题。

## 量化交易安全约束

- 策略代码不得直接调用券商 API。策略只能生成交易意图，必须经过风控、OMS 和统一券商网关。
- OMS 下单失败绝对不能自动重试。任何下单接口超时、网络异常、券商返回未知状态或系统异常，都必须进入 `failed` / `unknown` 等人工可审计状态，并通过订单查询、对账和人工确认处理，禁止用自动重试来“补偿”失败。
- 重大消息面或宏观事件可以触发快速风险退出设计，目标是在事件发布或行情确认后 10 分钟内注入策略/风控上下文；该链路只允许暂停开仓、降低仓位或清理风险暴露，不能绕过策略、风控和 OMS，不能用于自动加仓。
- LLM 不得直接触发实盘下单。LLM 只能用于研究、解释、生成候选因子或候选策略代码；候选结果必须经过沙箱、回测、人工审核和风控配置。
- 默认保守处理订单类型。未确认前，不默认使用市价单、融资融券、杠杆、期权、做空、盘前盘后交易。
- 风控必须优先于收益目标。任何绕过风控的实现都视为错误实现。
- 本地订单、成交、持仓和资金记录必须能与券商账户对账。
- 发现账户状态、订单状态或持仓不一致时，默认暂停相关策略并提示人工确认。

## 用户协作与解释规范

- 用户熟悉 Python 技术栈，但正在学习量化交易；解释量化概念时要通俗说明目的、风险和工程落点。
- 引入新概念时先讲“它解决什么问题”，再讲实现细节。例如 OMS、风控、滑点、回测、因子、IC、对账、交易日历等。
- 复杂金融术语不要只给缩写。第一次出现时补一句中文解释。
- 对不确定的券商 API 行为、交易规则、市场规则，不要猜测为事实；应标记为待确认，必要时查看官方文档。
- 代码注释以中文为主。关键业务规则、交易状态机、风控判断、券商适配器差异、并发和资源生命周期必须写清楚。

## 文档语言规范

- 与用户对话：中文。
- `docs/` 文档：简体中文，标准英文术语可保留英文。
- 代码注释：中文为主，标准英文术语、协议名、API 名保持原文。
- 用户可见 UI 文案：默认简体中文。
- 券商申请材料如需繁体中文或指定格式，单独按券商要求处理，不影响内部 `docs/` 的简体中文规范。
