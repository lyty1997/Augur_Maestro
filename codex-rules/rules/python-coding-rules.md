# Python 编码规范

## 基本原则

- Python 代码以清晰、可测试、可审计为优先。
- 业务边界优先于技巧性抽象。
- 金额、价格、数量、手续费、汇率等金融数值优先使用 `Decimal`，不得随意使用浮点数承载关键交易计算。
- 时间字段使用带时区的 `datetime`，明确交易所时区、系统时区和数据源时区。

## 类型与模型

- 公共函数、服务接口、数据模型应写类型注解。
- 配置、API 入参、外部响应解析优先使用 Pydantic、dataclass 或明确 DTO。
- 账户、资金、持仓、订单、成交、交易意图等模型必须与设计文档一致。
- 不在业务代码中混用券商原始字段和内部标准字段；应在适配器边界完成转换。

## 错误处理

- 券商错误、网络错误、超时、解析错误必须保留上下文并脱敏。
- 不吞异常；若转换为业务错误，需要保留可排查原因。
- 下单相关错误不得自动重试。
- 未知状态必须显式表示为 `unknown` 或等价状态，等待查询、对账或人工确认。

## 资源与安全

- HTTP 客户端、数据库连接、WebSocket、文件句柄、后台任务必须有生命周期管理。
- 不使用 `eval`、`exec` 执行不可信输入。
- 调用 subprocess 时优先传参数列表，不使用 `shell=True` 处理用户输入。
- SQL 必须参数化，不拼接用户输入。
- 日志不得包含密钥、token、真实账户号、身份证明、完整资金隐私。

## 测试

- 新增业务逻辑应补充单元测试。
- 券商适配器测试默认使用夹具、mock、dry-run 或模拟服务。
- 测试夹具必须脱敏。
- 涉及金额、数量、状态机、风控、边界时间的测试要覆盖异常分支。
