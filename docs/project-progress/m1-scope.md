## 1. 项目形态

M1 是本地单用户研究系统，不是托管交易 SaaS，也不是替用户管理资金的平台。

M1 的核心目标是建立一个可复现、可审计、可替换数据源的 A 股研究回测闭环。它帮助用户学习和验证量化研究流程，但不产生真实交易动作。

默认运行形态：

- 本地 Python 环境。
- 本地 PostgreSQL。
- Typer CLI 作为主入口。
- FastAPI 只保留健康检查和后续只读查询入口。
- M1 第一条关键路径先跑通 CLI；React 控制台后置，若实现也只做只读展示。

## 2. M1 核心目标

M1 的完成定义：

1. 能建立用户确认过的 A 股主题研究股票池。
2. 能通过 `DataProvider` 抽象从 AKShare 导入历史日线数据。
3. 能通过第二数据源对关键字段做交叉验证。
4. 能把数据标准化写入 PostgreSQL。
5. 能执行数据质量检查，并阻止不合格数据进入回测结论。
6. 能按时间顺序切分建模、回测和模拟留出区间。
7. 能并行最小验证 Qlib 和 VectorBT，再选择最适合的主执行器。
8. 能保存回测指标、权益曲线、虚拟订单、虚拟成交和归因结果。
9. 能生成 Markdown 报告，清楚写出数据来源、参数、结果、归因和风险提示。
10. README 能让用户从空环境按命令复现一次完整流程。

M1 的系统价值不是收益率，而是把研究链路做成可靠账本。回测结果只用于工程验证和研究探索，不能直接作为实盘建议。

## 3. M1 确认要做

### 3.1 数据线

- 定义 `DataProvider` 抽象。
- 实现 `AkshareDataProvider`。
- 导入 A 股标的信息、交易日历和日线行情。
- 同时保存不复权 `none` 和前复权 `qfq` 数据。
- 记录数据源、Provider 版本、配置哈希、导入批次、入库时间和原始数据哈希。
- 引入 BaoStock 作为第二数据源做关键字段交叉验证；Tushare Pro 或本地 CSV/Parquet 作为后续备用方案。
- 建立数据质量检查批次和问题表。
- 数据质量问题至少覆盖字段缺失、重复记录、交易日缺失、价格关系异常、成交量/成交额异常和复权序列异常。

### 3.2 股票池线

- 建立 `research_universes` 和 `research_universe_members`。
- 建立 `universe_candidate_runs` 和 `universe_candidates`。
- 支持人工、规则、AKShare 概念板块或 AI 辅助生成候选股票。
- 允许 AI 扩展候选清单，但所有 AI 扩展结果仍需人工确认。
- 候选必须有代码、名称、主题、来源、理由和人工确认状态。
- 未经人工确认的候选不能进入正式研究股票池。
- 候选理由不能只写关键词命中，必须说明真实关联依据。
- 正式研究池需要排除表现异常、容易被大资金操纵的小票股。
- 换手率低、成交额低、长期停牌、流动性差的标的不进入正式研究池。
- 第一版保守阈值：最小上市天数 250、20 日均成交额不低于 5000 万元、20 日均换手率不低于 0.5%、近 250 个交易日停牌天数不超过 20、总市值不低于 50 亿元。

M1 主题范围：

- 算力。
- 电力。
- 半导体。
- 人工智能。
- 芯片。
- 电池。
- 通信。
- 银行。
- CPO。
- 机器人。
- 存储。
- 有色金属。
- 影视院线。

### 3.3 研究任务线

- 建立 YAML 研究任务配置。
- 建立研究任务状态记录。
- 保存数据区间、股票池、切分方式、复权方式、配置哈希和失败原因。
- 默认导入最近 10 年数据。
- 最近 1 年作为模拟留出数据。
- 其余数据按时间顺序 8:2 切分为建模数据和回测数据。
- 切分不能随机打乱，避免未来函数。

### 3.4 回测线

- 先并行最小验证 Qlib 和 VectorBT，再选择最适合 RobustQuant 的主执行器。
- 回测必须读取 PostgreSQL 标准表，不能直接访问 AKShare。
- 回测配置必须保存初始资金、手续费、滑点、调仓规则、复权方式和配置哈希。
- 回测结果至少保存总收益、年化收益、最大回撤、波动率、夏普比率、胜率和换手率。
- 回测虚拟订单和虚拟成交必须明确标记为回测数据，不能复用未来真实 OMS 表。

### 3.5 归因和报告线

- 归因至少覆盖标的、主题、交易成本和主要回撤片段。
- 若因子和事件窗口尚未进入第一轮实现，报告中必须标记为未覆盖，不得假装已有结论。
- 报告必须包含风险提示：
  - M1 结果只用于研究探索。
  - AKShare 不是最终生产真相源。
  - 主题股票池可能有幸存者偏差。
  - 回测不能代表未来收益。
  - 任何策略都不能直接进入实盘。

### 3.6 工程线

- Python 项目使用 `venv` + `requirements.txt` 起步。
- 数据库访问使用 SQLAlchemy 2.x。
- 数据库迁移使用 Alembic。
- CLI 使用 Typer。
- 所有 CLI 命令必须有 `--help`。
- `.env` 保存本地敏感配置，不进入 Git。
- `configs/*.yaml` 保存研究配置，可进入 Git。
- `outputs/` 保存运行产物，不进入 Git。
- `rq_core` 不依赖 FastAPI、Typer、SQLAlchemy、AKShare、Qlib 或 VectorBT。

## 4. M1 明确不做

M1 不做以下事项：

- 真实下单。
- 真实撤单。
- 真实条件单、预埋单、止盈止损、触发单。
- 真实订单查询、真实成交查询、真实持仓查询和真实资金查询。
- 真实 OMS、真实风控拦截、真实账户模型和真实对账模型。
- miniQMT 真实交易接入。
- 盈立 OpenAPI 正式交易接入。
- 盈立 OpenAPI 真实下单、改单、撤单运行。
- 高频交易、盘口抢单、Level-2 行情。
- 多账户资金分配。
- 多策略订单合并。
- Redis、Celery、Dramatiq 等后台任务队列。
- LLM 自动生成并运行可执行策略代码。
- LLM 或消息面直接触发交易。
- React 中的真实买入、卖出、撤单按钮。

## 5. M1 设计预留但不实现

以下能力可以在文档和接口中预留，但不能在 M1 调用真实券商接口：

- `risk_kernel`。
- `oms_kernel`。
- `broker_kernel`。
- `BrokerTradingAdapter` 占位接口和 `TradingGateway` 安全门面。
- 只读券商能力清单。
- 交易意图模型。
- 订单状态草案。
- 交易时间检查接口。
- 账户、持仓、资金、成交和对账模型草案。

预留接口不能给用户造成“已经可以实盘”的误解。所有配置默认必须关闭真实交易能力。

## 6. 安全硬边界

以下规则对 M1 和后续版本都有效：

- 策略不得直接调用券商 API。
- LLM 不得直接触发实盘下单。
- 任何会向券商提交委托、撤单、条件单、预埋单或改变券商订单状态的调用，都按真实交易处理。
- 不允许用真实下单或真实撤单接口做连通性测试。
- 非交易时间不得向券商提交真实挂单。
- OMS 下单失败、超时、网络异常或未知状态时，不允许自动重试。
- 发现账户、订单、持仓或资金不一致时，默认暂停相关策略并提示人工确认。
- 券商 API Key、Secret、token、密码、真实账号、真实身份证明和真实资金隐私不得写入代码、文档、日志或报告。

## 7. M1 验收标准

M1 验收时至少检查：

| 类别 | 验收项 |
|---|---|
| 环境 | 新环境能按 README 创建虚拟环境、安装依赖、启动 PostgreSQL |
| 数据库 | Alembic 能从空库创建 M1 全部表 |
| CLI | 所有命令都有 `--help`，失败返回非 0 退出码 |
| 股票池 | 候选股票必须人工确认后才能进入正式股票池 |
| 数据导入 | 能导入主题股票池最近 10 年日线，不足 10 年从可得首日开始 |
| 数据质量 | 质量检查能输出批次、问题、严重级别和阻断状态 |
| 回测 | 至少一个成熟回测库能跑通 A 股日频最小策略 |
| 归因 | 能回答主要收益和亏损来自哪些标的、主题、成本或回撤片段 |
| 报告 | Markdown 报告包含数据源、复权、参数、结果、归因和风险提示 |
| 安全 | 代码中不存在真实券商下单、撤单或条件单调用 |
| 隐私 | 不提交任何券商密钥、账号隐私或资金隐私 |

## 8. 范围变更流程

任何范围变更都按以下步骤处理：

1. 在本文档写明新增或变更项。
2. 标明为什么需要变更。
3. 说明影响哪些专项文档、数据模型、接口或验收标准。
4. 若涉及交易、风控、账户、订单、券商适配器或部署方式，先更新对应设计文档。
5. 用户确认后再进入编码。

## 9. M2 和 M3 摘要

M2 目标是模拟盘与风控预演，原则是“像实盘一样审计，但不碰真实资金”。M2 可以实现策略运行时、交易意图、基础风控、模拟 OMS、虚拟账户和只读监控。

M3 目标是受控实盘交易闭环，前提是 miniQMT 或其他券商 API 行为已经确认，且真实 OMS、风控、交易时间检查、账户/标的白名单、人工暂停、撤单、对账和日志设计全部落地。实盘第一批优先选择价格较低、流动性好的宽基 ETF 验证链路，个股实盘后置；模拟盘可以更自由地覆盖个股和主题股票池。

M3 进入任何真实下单前，都必须重新更新交易链路设计并经用户确认。

盈立 OpenAPI 申请材料需要源码截图时，该事项按券商申请材料处理，不作为 RobustQuant 开发交付。开发侧只按 [yingli-openapi-reference.md](../backend/clients/yingli-openapi-reference.md) 解析官方 PDF，并且不允许真实提交委托。
