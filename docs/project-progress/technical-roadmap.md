## 2. 总体原则

- 先设计，后编码；涉及交易、风控、账户、订单、数据模型和部署方式的变更，必须先更新设计文档。
- M1 先做 A 股研究回测闭环，不接真实交易。
- 策略、因子、回测、风控、OMS、券商适配器必须解耦。
- 研究数据先可用、可复现、可替换，再追求完整。
- 实盘能力必须可审计、可暂停、可撤单、可对账、可复盘。
- 非交易时间不得向券商提交真实下单或挂单请求。
- miniQMT 和盈立 OpenAPI 只读查询不构成真实交易；任何会向券商提交委托、撤单、条件单、预埋单或触发券商侧订单状态变化的调用，都按真实交易处理。
- 消息面和 AI 不能仅凭关键词、同名或概念碰瓷选股交易。
- LLM 只能进入研究辅助链路，不能直接触发真实下单。
- OMS 下单失败或未知状态绝不自动重试。

## 3. 已确认技术方向

### 3.1 M1 范围

- 市场：A 股。
- 主线：研究回测闭环。
- 交易：只做统一券商接口抽象和占位，不接真实交易。
- 历史数据源：默认使用 AKShare 作为研究启动源。
- 第二数据源：M1 第一版选择 BaoStock，至少做关键字段抽样交叉验证。
- 股票池：M1 只导入用户熟悉主题相关标的，不先导入全 A 股；AI 可以辅助生成候选股票池，但必须经用户确认。
- 股票池过滤：排除 ST、长期停牌、低成交额、低换手率、成交稀疏和疑似容易被大资金操纵的小票股；第一版保守阈值写入配置。
- 数据范围：默认导入最近 10 年，最近 1 年作为模拟数据，之前数据按时间顺序 8:2 切分为建模数据和回测数据。
- 数据库：PostgreSQL。
- 部署：本地部署，优先使用 Docker Compose，本机 PostgreSQL 作为备选。
- Python 项目管理：`venv` + `requirements.txt` 起步。
- 后端：FastAPI。
- 前端：React 是技术方向，但 M1 主链路先跑通 CLI；React 控制台在最小闭环后可选补。
- 回测引擎：并行完成 Qlib 和 VectorBT 最小验证，再选择 M1 主执行器或明确两者分工。
- 任务入口：第一版使用 YAML 配置文件 + Typer CLI 命令，CLI 必须提供完整 `--help`，README 必须写清运行顺序。
- 数据库访问：SQLAlchemy 2.x + Alembic。
- 报告格式：第一版使用 Markdown。
- 输出和中间产物：默认写入 `outputs/`，不进入代码库。
- 核心架构：拆分 kernel，保证高内聚、低耦合。
- Kernel 接口类：按 [kernel-interfaces.md](../backend/kernel/kernel-interfaces.md) 定义 DTO、Protocol 和服务接口；`rq_core` 不依赖 FastAPI、Typer、SQLAlchemy、AKShare、Qlib 或 VectorBT。

### 3.2 暂不进入 M1

- 真实下单。
- 条件单、止盈止损、触发单、分批下单等高级订单能力。
- 多市场同时研究。
- 高频、盘口、Level-2 行情。
- LLM 自动生成并运行策略代码。
- 盈立 OpenAPI 正式接入。源码截图材料属于券商申请事项，不作为开发任务；开发侧只解析官方网页手册，不执行真实交易接口。

## 4. 总体模块路线

RobustQuant 按七条技术主线推进。

### 4.1 数据线

目标：建立 A 股研究数据链路，让研究和回测读取内部标准表，而不是直接依赖 AKShare。

M1 任务：

- 定义 `DataProvider` 抽象。
- 实现 `AkshareDataProvider`。
- 建立 PostgreSQL 标准表：标的信息、交易日历、日线行情、导入批次。
- 建立研究股票池表：股票池定义、股票池成员、主题标签。
- 建立 AI/规则候选股票池生成和人工确认流程。
- 同时保存不复权和前复权日线数据。
- 建立基础数据质量检查。
- 接入第二数据源做关键字段抽样交叉验证。
- 记录数据源、依赖版本、导入批次、复权方式和入库时间。

后续任务：

- 扩展更多交叉验证源和备用数据源。
- miniQMT 可用后实现 `MiniQMTDataProvider`。
- 实盘前用券商数据复核研究期数据。

专项路线见 [data-source-roadmap.md](../backend/data/data-source-roadmap.md)。

### 4.2 研究与回测线

目标：让一个策略想法可以从配置任务进入数据准备、因子计算、回测、报告和人工审核。

M1 任务：

- 定义研究任务配置格式。
- 建立 AI/规则辅助候选股票池和人工确认流程。
- 建立股票池过滤规则。
- 建立基础技术因子计算。
- 选择并接入成熟回测库。
- 增加归因分析，拆解收益和亏损来源。
- 输出 Markdown 研究报告。
- 记录实验参数、数据版本、回测结果和失败原因。

后续任务：

- 支持 walk-forward 分段训练和验证。
- 支持模型训练，例如 LightGBM、XGBoost。
- 支持模拟盘或离线 replay。
- 增加事件窗口分析。

### 4.3 后端 API 线

目标：用 FastAPI 提供稳定的系统入口，但不让 API 层承载交易或策略核心逻辑。

M1 任务：

- 搭建 FastAPI 基础工程。
- 第一版先提供健康检查和基础查询接口。
- 数据导入、质量检查和最小回测先通过 CLI 执行。
- 后续再提供数据导入任务查询接口、研究任务创建查询接口、回测报告查询接口。
- 建立配置读取和环境变量管理。

后续任务：

- 增加用户登录和权限。
- 增加策略状态、模拟盘状态、风控配置接口。
- 增加订单、成交、持仓、对账查询接口。

### 4.4 前端控制台线

目标：用 React 提供研究和监控入口。前端只负责展示、配置和人工干预，不承载核心业务判断。

M1 任务：

- 查看数据导入批次。
- 查看数据质量检查结果。
- 创建和查看研究任务。
- 查看回测报告。

后续任务：

- 策略启停。
- 风控参数配置。
- 模拟盘监控。
- 订单、成交、持仓、对账监控。
- 人工暂停、撤单和异常确认入口。

### 4.5 风控与 OMS 线

目标：为后续实盘建立安全边界。M1 先做模型和接口设计，不接真实交易。

M1 任务：

- 定义交易意图模型。
- 定义订单请求、订单状态、成交记录、持仓快照、资金快照模型。
- 定义风控检查结果模型。
- 定义 OMS 状态机草案。
- 明确下单失败、超时、未知状态绝不自动重试。

后续任务：

- 实现基础事前风控：单笔金额、单票仓位、交易次数、黑名单、价格偏离。
- 实现 OMS 订单生命周期。
- 实现撤单和对账。
- 设计条件单、高级订单和本地托管触发规则。

### 4.6 券商适配线

目标：让策略和 OMS 不直接依赖具体券商 SDK。

M1 任务：

- 定义 `BrokerTradingAdapter` 抽象和 `TradingGateway` 统一交易门面。
- 实现占位适配器和模拟适配器。
- 梳理 miniQMT 运行环境、权限、登录、行情和交易接口待确认清单。

后续任务：

- 实现 miniQMT 账户查询、持仓查询、订单查询、成交查询。
- 在充分确认后接入下单和撤单。
- 盈立 OpenAPI 若申请通过，再设计港股和美股适配器。

### 4.7 AI/LLM 研究辅助线

目标：让 LLM 成为研究助手，而不是交易执行者。

M1 任务：

- 暂不让 LLM 生成可执行实盘代码。
- 可前置用于候选股票池生成、报告解释、研究任务拆解、候选因子说明等辅助能力设计。
- LLM 或消息面不能仅凭关键词、同名或概念碰瓷选股交易。

后续任务：

- LLM 将自然语言研究目标转换为结构化配置。
- LLM 辅助解释回测结果和风险。
- LLM 生成候选因子或策略草案，但必须经过沙箱、回测、人工审核。

## 5. 里程碑路线图

### 设计基线：设计与工程底座

目标：把系统边界和安全原则定住，形成可以开始编码的设计基础。

主要交付：

- 系统架构设计。
- 总体技术路线规划。
- 历史数据源技术路线规划。
- 研究回测 Agent 设计。
- 待决策清单。
- Git 仓库和基础忽略规则。

验收标准：

- M1 范围、数据源、技术栈、回测方向已确认。
- 涉及实盘的模块只做抽象设计，不实现真实交易。
- 所有新增能力能在文档中找到设计依据。

### M1：A 股研究回测闭环

目标：用 AKShare、PostgreSQL、CLI、成熟回测库和 Markdown 报告，跑通用户熟悉主题股票池的 A 股研究回测闭环。M1 不接真实交易。

主要交付：

- PostgreSQL 数据表。
- Alembic 数据库迁移。
- `DataProvider` 抽象。
- `AkshareDataProvider`。
- `BaostockDataProvider` 作为第二数据源 Provider。
- 主题股票池配置。
- AI/规则候选股票池生成和人工确认流程。
- 低流动性、长期停牌、异常小票过滤规则。
- 数据导入批次记录。
- 数据质量检查报告。
- 第二数据源关键字段交叉验证报告。
- 研究任务配置。
- 股票池过滤器。
- 最近 1 年模拟数据留出，以及此前数据 8:2 建模/回测切分。
- 基础技术因子。
- Qlib 和 VectorBT 并行最小验证，至少各跑通一个同口径 A 股日频回测样例，再选择主执行器。
- 归因分析结果，包括标的、主题、因子、时间窗口、事件窗口、交易成本和主要回撤来源。
- Markdown 回测报告。
- CLI 命令和 README 运行说明。

验收标准：

- 能导入用户熟悉主题股票池最近 10 年日线数据；如果历史数据不足 10 年，从可获得数据首日开始。
- 能同时保存不复权和前复权数据。
- 数据导入失败和部分失败可追踪。
- 第二数据源关键字段差异可以在质量报告中追踪。
- 长期停牌、低成交额、低换手率和异常小票默认不会进入正式研究股票池。
- 策略和回测代码不直接调用 AKShare。
- 每个 CLI 命令都有 `--help`。
- 能运行一个 A 股日频策略回测。
- 报告包含数据范围、数据源、复权方式、参数、收益、回撤、换手率、归因分析和风险提示。
- 回测结果可复现。

### M2：模拟盘与风控预演

目标：在不接真实交易的前提下，把通过 M1 研究审核的策略放入模拟运行环境，验证策略运行、交易意图、风控、模拟 OMS 和人工干预流程。

主要交付：

- 研究 Agent 编排和实验记录。
- 策略运行时。
- 交易意图模型。
- 基础风控引擎。
- 模拟 OMS。
- 虚拟订单、成交、持仓。
- React 控制台第一版，用于查看研究、模拟盘、风控和异常状态。

验收标准：

- 每次研究任务都有状态、参数、数据版本、产物和错误原因。
- Agent 输出策略草案后进入人工审核，不能进入实盘。
- 策略只能生成交易意图。
- 交易意图必须经过风控。
- 模拟订单状态和风控拒绝原因可审计。

### M3：受控实盘交易闭环与扩展

目标：在 miniQMT 权限、环境和 API 行为确认后，小心接入真实交易链路；A 股实盘闭环稳定后，再扩展多券商、多市场和高级订单能力。

前置条件：

- miniQMT 运行环境确认。
- 账户查询、持仓查询、订单查询、成交查询能力确认。
- 下单、撤单 API 行为确认。
- 风控、OMS、日志、对账和人工暂停入口完成。
- 若使用盈立 OpenAPI，必须先完成官方网页手册字段确认；用户已确认未得到盈立明确确认 UAT / sandbox 不产生真实委托前，禁止真实提交下单、改单或撤单请求。

主要交付：

- `MiniQMTTradingAdapter`。
- 账户、持仓、订单、成交查询。
- 下单、撤单。
- 对账任务。
- Web 监控和人工暂停。
- 条件单、止盈止损、触发单、分批下单等高级订单能力设计与实现。
- 盈立 OpenAPI 港股、美股适配器，在申请通过、官方网页手册字段确认和接口行为确认后再设计。
- 多市场交易日历、多币种账户视图和更完整的数据源接入。

验收标准：

- 所有真实订单都有策略来源、风控结果、OMS 状态和券商返回。
- 下单失败或未知状态不自动重试。
- 发现账户、订单或持仓不一致时暂停相关策略并提示人工确认。
- 新市场不破坏现有 A 股链路。
- 高级订单不绕过风控和 OMS。
- LLM 不进入真实下单链路。

## 6. 细分实施计划

细分实施计划使用 `M1.x`、`M2.x`、`M3.x` 表达，表示对应顶层里程碑内部的批次，不再增加新的顶层阶段。每个批次完成后都应更新 README、命令帮助、设计文档和必要的测试。

### 6.1 M1 细分：A 股研究回测闭环

M1 的原则是先让研究链路可复现，再补体验。M1 结束时必须能用一条明确命令链从股票池、数据导入、质量检查、回测到报告跑通。

#### M1.0：工程底座和开发规范

目标：建立可以安全迭代的工程骨架。

主要交付：

- Python `venv` + `requirements.txt`。
- `requirements-dev.txt`、`requirements-dev.lock.txt` 和 `pyproject.toml`，统一本地与 CI 的测试、覆盖率、lint、typecheck 配置。
- 基础目录结构：`src/rq_core/`、`src/cli/`、`src/backend/`、`configs/`、`tests/`、`outputs/`。
- `.env.example` 和本地配置读取规则。
- 代码格式化、lint、typecheck、pytest 基础命令。
- GitHub Actions CI，push 和 pull request 时运行 `ruff check`、`ruff format --check`、`mypy`、密钥扫描、依赖漏洞扫描、交易专项安全静态检查、Markdown 文档检查和 `pytest` 覆盖率门禁。
- PR 模板和 CODEOWNERS，提醒交易安全、文档先行和敏感信息红线。
- README 第一版，写清本地启动、测试和 CLI 查看帮助的方法。

验收标准：

- 新环境能按 README 创建虚拟环境并安装依赖。
- `outputs/`、`.env`、数据库本地数据文件不进入代码库。
- 基础测试命令可以运行，即使初期只有少量冒烟测试。
- 第一版测试覆盖率阈值不低于 60%，后续随模块增加逐步提高。
- 依赖漏洞扫描、交易专项安全静态检查和 Markdown 文档检查必须通过。
- CI 不触碰真实券商接口，只运行 dry-run、fake、fixture 或纯单元测试。

#### M1.1：PostgreSQL 和迁移底座

目标：让数据库结构可版本化、可重建、可解释。

主要交付：

- 本地 PostgreSQL 启动方案，优先 Docker Compose，本机服务作为备选。
- SQLAlchemy 2.x 数据访问基础。
- Alembic 迁移机制。
- M1 核心表迁移：标的信息、交易日历、日线行情、导入批次、数据质量检查、股票池、研究任务、回测结果、归因分析、报告产物。

验收标准：

- 空库可以通过迁移创建完整 M1 表结构。
- 表字段、约束和状态枚举与 [data-model.md](../backend/data/data-model.md) 一致。
- 迁移失败不会静默吞掉错误。

#### M1.2：Kernel 接口和 CLI 骨架

目标：先定内核协作边界，再接具体数据源和回测库。

主要交付：

- 按 [kernel-interfaces.md](../backend/kernel/kernel-interfaces.md) 落地 DTO、Protocol 和服务接口。
- Typer CLI 基础入口。
- CLI 命令分组：`universe`、`data`、`quality`、`research`、`backtest`、`attribution`、`report`。
- FastAPI 健康检查接口，暂不承载核心研究流程。

验收标准：

- 每个 CLI 命令都有 `--help`。
- `rq_core` 不依赖 FastAPI、Typer、SQLAlchemy、AKShare、Qlib 或 VectorBT。
- CLI 只作为入口层调用 kernel 服务，不沉淀核心业务规则。

#### M1.3：主题股票池和人工确认

目标：只导入用户熟悉主题相关标的，避免一开始把全 A 股复杂度拉满。

主要交付：

- 主题配置文件，覆盖算力、电力、半导体、人工智能、芯片、电池、通信、银行、CPO、机器人、存储、有色金属、影视院线。
- AI/规则辅助候选股票池生成流程。
- 候选标的必须使用“代码 + 名称”表达。
- 候选理由、来源、主题、置信度和人工确认状态入库。

验收标准：

- 未经人工确认的候选标的不能进入正式研究股票池。
- 不能仅凭关键词或同名关系把新闻、概念和股票强行关联。
- 可以导出待确认清单，方便人工逐只检查。

#### M1.4：AKShare、第二数据源导入和质量检查

目标：建立可替换的数据接入链路，而不是让策略直接调用 AKShare。

主要交付：

- `AkshareDataProvider`。
- `BaostockDataProvider` 作为第二数据源 Provider；Tushare Pro 或本地 CSV/Parquet 只作为备用方案。
- A 股交易日历、标的信息、日线行情导入。
- 最近 10 年数据导入，上市不足 10 年的标的从可获得数据首日开始。
- 同时保存不复权和前复权数据。
- 导入批次、依赖版本、数据源、复权方式和入库时间记录。
- 字段完整性、时间完整性、重复记录、价格异常、复权序列断裂等质量检查。
- 第二数据源关键字段抽样交叉验证，至少覆盖收盘价、成交量、成交额和停牌/缺失情况。

验收标准：

- 策略、因子和回测代码中不出现 AKShare 直接调用。
- 导入失败、部分失败和质量检查失败都有状态和错误原因。
- 数据质量未通过的数据不能被标记为可用于回测。
- 单一数据源无法交叉验证时，质量报告必须降级为 warning 或 failed，并记录原因。

#### M1.5：数据切分、基础因子和回测库验证

目标：跑通最小研究实验，验证成熟回测库能适配 A 股日频数据。

主要交付：

- 最近 1 年模拟数据留出。
- 此前历史数据按时间顺序 8:2 切分为建模数据和回测数据。
- 基础技术因子，例如动量、均线、波动率、成交额过滤。
- Qlib 和 VectorBT 并行最小验证。
- 基于同一数据、同一股票池、同一成本假设选择 M1 主回测执行器，或明确两者分工。
- 基础归因分析，先覆盖标的贡献、主题贡献、交易成本贡献和主要回撤片段。

验收标准：

- Qlib 和 VectorBT 都要完成最小验证；若其中一个因环境或适配限制失败，必须记录失败原因和替代方案。
- 回测使用标准化 PostgreSQL 数据，不直接读外部数据源。
- 回测配置、数据窗口、参数和结果可复现。
- 能回答“主要收益来自哪些标的/主题”和“主要亏损来自哪些标的/时间窗口/交易成本”。

#### M1.6：研究任务、报告和复盘闭环

目标：把一次研究从临时脚本变成可追踪任务。

主要交付：

- 研究任务配置 YAML。
- 回测结果入库。
- 归因分析结果入库。
- Markdown 报告输出到 `outputs/`。
- 报告包含数据范围、数据源、复权方式、参数、收益、回撤、换手率、归因分析、风险提示和失败原因。
- README 增加完整端到端示例命令链。

验收标准：

- 一条命令链可以从已确认股票池生成第一份研究报告。
- 中间产物和报告不进入代码库。
- 报告不得暗示可以直接进入实盘，只能给出研究结论和下一步审核建议。

#### M1.7：M1 收口和是否补 React 控制台

目标：确认 M1 是否已经达到“可以继续做模拟盘”的条件。

主要交付：

- M1 验收清单。
- 关键测试和样例数据。
- 文档更新：README、架构、数据模型、接口类。
- 可选 React 只读控制台，用于查看导入批次、质量检查、研究任务和报告。

验收标准：

- M1 端到端流程在本地可复现。
- 未完成项进入 [open-decisions.md](../architecture/open-decisions.md) 或后续任务清单。
- 若补 React 控制台，前端只读展示，不引入真实交易和策略启停能力。

### 6.2 M2 细分：模拟盘与风控预演

M2 的原则是“像实盘一样审计，但不碰真实资金”。M2 产生的是模拟订单和模拟持仓，不向券商提交任何真实委托。

#### M2.1：研究 Agent 编排

目标：把 M1 的手工命令链升级为可追踪的研究任务编排。

主要交付：

- 研究计划器。
- 数据窗口规划器。
- 因子评估流程。
- 实验记录和对比视图。
- 人工审核状态流转。

验收标准：

- 每次研究任务都有状态、参数、数据版本、产物和错误原因。
- Agent 输出策略草案后只能进入人工审核，不能自动进入模拟盘或实盘。

#### M2.2：策略运行时和离线 replay

目标：验证策略在按时间推进时是否能稳定生成交易意图。

主要交付：

- 策略运行时。
- 离线 replay 数据流。
- 策略启停状态。
- 交易意图模型。

验收标准：

- 策略只能生成交易意图，不能直接生成券商订单。
- 每个交易意图都能追溯到策略、参数、数据窗口和触发原因。

#### M2.3：基础风控引擎

目标：让所有交易意图先过风控，再进入模拟 OMS。

主要交付：

- 单笔金额、单票仓位、交易次数、黑名单、价格偏离等基础规则。
- 交易时间检查。
- 消息面和宏观事件的风险上下文输入。
- 风控通过、拒绝、降级、暂停等结果模型。

验收标准：

- 非交易时间不得产生可执行订单，只能产生待审计交易意图。
- 风控拒绝必须记录规则、输入、结果和解释。
- 消息面不能绕过策略、风控和 OMS。

#### M2.4：模拟 OMS 和虚拟账户

目标：在模拟环境中验证订单生命周期、持仓变化和异常处理。

主要交付：

- 模拟 OMS 状态机。
- 虚拟订单、成交、持仓、资金。
- 手续费、印花税、滑点和涨跌停约束。
- 下单失败、超时、未知状态的模拟处理。

验收标准：

- 模拟订单状态可追踪。
- 下单失败或未知状态不自动重试。
- 虚拟持仓和资金可以按交易日复盘。

#### M2.5：React 监控和人工干预入口

目标：让模拟盘和风控状态可看、可暂停、可复盘。

主要交付：

- 研究任务列表和报告查看。
- 模拟盘状态看板。
- 风控拒绝记录。
- 策略暂停和恢复入口。
- 异常确认入口。

验收标准：

- 前端不承载核心业务规则。
- 所有人工操作都有日志。
- UI 不提供真实下单入口。

### 6.3 M3 细分：受控实盘交易闭环与扩展

M3 的原则是“先查询、后交易；先小范围、后扩展”。进入任何真实下单能力前，必须重新更新交易链路、风控、OMS、券商适配器、账户和对账设计，并经用户确认。

#### M3.1：miniQMT / OpenAPI 能力确认和只读接入

目标：先确认券商接口真实行为，不直接下单。miniQMT 和盈立 OpenAPI 都必须先完成只读接入验证，再讨论任何真实交易能力。

主要交付：

- miniQMT 运行环境确认。
- 登录、重连、权限、行情、账户查询、持仓查询、订单查询、成交查询能力清单。
- 盈立 OpenAPI 若申请通过，先基于官方网页手册确认认证方式、行情权限、账户查询、持仓查询、订单查询和成交查询能力。
- `MiniQMTTradingAdapter` 只读能力。
- `YingLiOpenApiTradingAdapter` 只读能力，前提是 OpenAPI 申请和官方网页手册字段确认完成。
- 券商数据和 AKShare 数据差异报告。

验收标准：

- 不包含真实下单和撤单。
- 不调用任何可能形成委托、撤单、条件单、预埋单或券商侧订单状态变化的接口。
- 查询结果能与券商客户端人工核对。
- 接口异常、断线、权限不足都有明确错误状态。

#### M3.2：真实 OMS、风控和对账设计落地

目标：在真实交易前建立完整安全壳。

主要交付：

- 真实订单、成交、持仓、资金和对账表。
- OMS 状态机。
- 事前、事中、事后风控。
- 一键暂停。
- 日志和审计链路。

验收标准：

- 任何真实订单都有策略来源、交易意图、风控结果、OMS 状态和券商返回。
- 发现账户、订单或持仓不一致时，默认暂停相关策略并提示人工确认。

#### M3.3：受控下单和撤单闭环

目标：在极小范围内验证真实下单、撤单和状态同步。

主要交付：

- 下单接口。
- 撤单接口。
- 订单查询和成交查询同步。
- 人工确认开关。
- 最小资金和最小标的范围配置。
- `trading_enabled=false` 作为默认配置，只有显式开启并满足全部安全前置条件后才允许提交真实委托。

验收标准：

- 下单失败、超时或未知状态绝不自动重试。
- 非交易时间不得向券商提交真实下单或挂单请求。
- 不允许用真实下单、真实撤单或真实条件单接口做连通性测试。
- 真实交易必须可暂停、可撤单、可对账、可复盘。

#### M3.4：条件单和高级订单能力

目标：在真实闭环稳定后，再设计更复杂订单能力。

主要交付：

- 条件单、止盈止损、触发单、分批下单模型。
- 券商原生支持能力确认。
- 本地 OMS 托管触发方案，含断线恢复、重复触发防护和人工暂停。

验收标准：

- 高级订单不能绕过风控和 OMS。
- 本地托管条件单在系统不可用时必须进入保守状态。
- 所有触发条件、触发价格、撤单和成交结果可审计。

#### M3.5：多市场、多券商和数据增强

目标：在 A 股闭环稳定后扩展，而不是一开始扩大复杂度。

主要交付：

- 盈立 OpenAPI 港股、美股适配器，前提是申请通过并确认官方 API 行为。
- 多市场交易日历。
- 多币种账户视图。
- 更完整的行情、财务、公告、新闻和宏观数据源。
- LLM 研究助手增强。

验收标准：

- 新市场不破坏已有 A 股链路。
- 港股、美股盘前盘后、订单类型、行情权限和频率限制必须按 uSmart 官方网页手册确认。
- LLM 仍然不能进入真实下单链路。

## 7. 关键依赖关系

- 没有标准化数据表，就不要让策略或回测直接接数据源。
- 没有数据质量检查，就不要把回测结论标记为可模拟盘。
- 没有 OMS 和风控状态机，就不要接真实下单。
- 没有对账和人工暂停，就不要做自动交易。
- 没有券商官方接口行为确认，就不要实现真实交易适配器。
