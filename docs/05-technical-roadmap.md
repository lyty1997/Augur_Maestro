# RobustQuant 总体技术路线规划

版本：v0.1  
状态：部分确认  
最后更新：2026-05-13

## 1. 文档定位

本文档是 RobustQuant 的项目级技术路线规划，用来回答“先做什么、后做什么、各模块如何衔接、每个阶段做到什么程度才算完成”。

专项设计文档负责展开具体细节：

- 系统架构设计见 [01-architecture.md](./01-architecture.md)。
- 研究回测 Agent 设计见 [04-research-agent.md](./04-research-agent.md)。
- A 股历史数据源路线见 [06-data-source-roadmap.md](./06-data-source-roadmap.md)。
- 待决策事项见 [03-open-decisions.md](./03-open-decisions.md)。

## 2. 总体原则

- 先设计，后编码；涉及交易、风控、账户、订单、数据模型和部署方式的变更，必须先更新设计文档。
- 第一阶段先做 A 股研究回测闭环，不接真实交易。
- 策略、因子、回测、风控、OMS、券商适配器必须解耦。
- 研究数据先可用、可复现、可替换，再追求完整。
- 实盘能力必须可审计、可暂停、可撤单、可对账、可复盘。
- LLM 只能进入研究辅助链路，不能直接触发真实下单。
- OMS 下单失败或未知状态绝不自动重试。

## 3. 已确认技术方向

### 3.1 第一阶段范围

- 市场：A 股。
- 主线：研究回测闭环。
- 交易：只做统一券商接口抽象和占位，不接真实交易。
- 历史数据源：默认使用 AKShare 作为研究启动源。
- 数据库：PostgreSQL。
- 后端：FastAPI。
- 前端：React。
- 回测引擎：优先使用成熟库，不自研完整回测框架。

### 3.2 暂不进入第一阶段

- 真实下单。
- 条件单、止盈止损、触发单、分批下单等高级订单能力。
- 多市场同时研究。
- 高频、盘口、Level-2 行情。
- LLM 自动生成并运行策略代码。
- 盈立 OpenAPI 接入。

## 4. 总体模块路线

RobustQuant 按七条技术主线推进。

### 4.1 数据线

目标：建立 A 股研究数据链路，让研究和回测读取内部标准表，而不是直接依赖 AKShare。

第一阶段任务：

- 定义 `DataProvider` 抽象。
- 实现 `AkshareDataProvider`。
- 建立 PostgreSQL 标准表：标的信息、交易日历、日线行情、导入批次。
- 同时保存不复权和前复权日线数据。
- 建立基础数据质量检查。
- 记录数据源、依赖版本、导入批次、复权方式和入库时间。

后续任务：

- 接入 BaoStock、Tushare 或本地文件作为交叉验证源。
- miniQMT 可用后实现 `MiniQMTDataProvider`。
- 实盘前用券商数据复核研究期数据。

专项路线见 [06-data-source-roadmap.md](./06-data-source-roadmap.md)。

### 4.2 研究与回测线

目标：让一个策略想法可以从配置任务进入数据准备、因子计算、回测、报告和人工审核。

第一阶段任务：

- 定义研究任务配置格式。
- 建立股票池过滤规则。
- 建立基础技术因子计算。
- 选择并接入成熟回测库。
- 输出 Markdown 研究报告。
- 记录实验参数、数据版本、回测结果和失败原因。

后续任务：

- 支持 walk-forward 分段训练和验证。
- 支持模型训练，例如 LightGBM、XGBoost。
- 支持模拟盘或离线 replay。
- 增加事件窗口分析。

### 4.3 后端 API 线

目标：用 FastAPI 提供稳定的系统入口，但不让 API 层承载交易或策略核心逻辑。

第一阶段任务：

- 搭建 FastAPI 基础工程。
- 提供数据导入任务查询接口。
- 提供研究任务创建、查询、运行接口。
- 提供回测报告查询接口。
- 建立配置读取和环境变量管理。

后续任务：

- 增加用户登录和权限。
- 增加策略状态、模拟盘状态、风控配置接口。
- 增加订单、成交、持仓、对账查询接口。

### 4.4 前端控制台线

目标：用 React 提供研究和监控入口。前端只负责展示、配置和人工干预，不承载核心业务判断。

第一阶段任务：

- 查看数据导入批次。
- 查看数据质量检查结果。
- 创建和查看研究任务。
- 查看回测报告。

后续任务：

- 策略启停。
- 风控参数配置。
- 模拟盘监控。
- 订单、成交、持仓、对账监控。
- 人工暂停、撤单和异常确认入口。

### 4.5 风控与 OMS 线

目标：为后续实盘建立安全边界。第一阶段先做模型和接口设计，不接真实交易。

第一阶段任务：

- 定义交易意图模型。
- 定义订单请求、订单状态、成交记录、持仓快照、资金快照模型。
- 定义风控检查结果模型。
- 定义 OMS 状态机草案。
- 明确下单失败、超时、未知状态绝不自动重试。

后续任务：

- 实现基础事前风控：单笔金额、单票仓位、交易次数、黑名单、价格偏离。
- 实现 OMS 订单生命周期。
- 实现撤单和对账。
- 设计条件单、高级订单和本地托管触发规则。

### 4.6 券商适配线

目标：让策略和 OMS 不直接依赖具体券商 SDK。

第一阶段任务：

- 定义 `BrokerAdapter` 抽象。
- 实现占位适配器和模拟适配器。
- 梳理 miniQMT 运行环境、权限、登录、行情和交易接口待确认清单。

后续任务：

- 实现 miniQMT 账户查询、持仓查询、订单查询、成交查询。
- 在充分确认后接入下单和撤单。
- 盈立 OpenAPI 若申请通过，再设计港股和美股适配器。

### 4.7 AI/LLM 研究辅助线

目标：让 LLM 成为研究助手，而不是交易执行者。

第一阶段任务：

- 暂不让 LLM 生成可执行实盘代码。
- 可保留报告解释、研究任务拆解、候选因子说明等辅助能力设计。

后续任务：

- LLM 将自然语言研究目标转换为结构化配置。
- LLM 辅助解释回测结果和风险。
- LLM 生成候选因子或策略草案，但必须经过沙箱、回测、人工审核。

## 5. 分阶段路线图

### 阶段 0：设计与工程底座

目标：把系统边界和安全原则定住，形成可以开始编码的设计基础。

主要交付：

- 系统架构设计。
- 总体技术路线规划。
- 历史数据源技术路线规划。
- 研究回测 Agent 设计。
- 待决策清单。
- Git 仓库和基础忽略规则。

验收标准：

- 第一阶段范围、数据源、技术栈、回测方向已确认。
- 涉及实盘的模块只做抽象设计，不实现真实交易。
- 所有新增能力能在文档中找到设计依据。

### 阶段 1：A 股历史数据底座

目标：用 AKShare 跑通 A 股日线数据导入、标准化、落库和质量检查。

主要交付：

- PostgreSQL 数据表。
- `DataProvider` 抽象。
- `AkshareDataProvider`。
- 数据导入批次记录。
- 数据质量检查报告。

验收标准：

- 能导入至少 100 只 A 股最近 5 年日线数据。
- 能同时保存不复权和前复权数据。
- 数据导入失败和部分失败可追踪。
- 策略和回测代码不直接调用 AKShare。

### 阶段 2：最小研究回测闭环

目标：让一个配置化研究任务能完成股票池过滤、基础因子、回测和报告。

主要交付：

- 研究任务配置。
- 股票池过滤器。
- 基础技术因子。
- 成熟回测库接入。
- Markdown 回测报告。

验收标准：

- 能运行一个 A 股日频策略回测。
- 报告包含数据范围、数据源、复权方式、参数、收益、回撤、换手率和风险提示。
- 回测结果可复现。

### 阶段 3：研究 Agent 编排

目标：把研究流程从单次脚本升级为可追踪的 Agent 编排任务。

主要交付：

- 研究计划器。
- 数据窗口规划器。
- 因子评估流程。
- 实验记录库。
- 人工审核状态。

验收标准：

- 每次研究任务都有状态、参数、数据版本、产物和错误原因。
- Agent 输出策略草案后进入人工审核，不能进入实盘。
- 可以对比多次实验结果。

### 阶段 4：模拟盘与风控预演

目标：在不接真实交易的前提下验证策略运行、交易意图、风控和虚拟订单流程。

主要交付：

- 策略运行时。
- 交易意图模型。
- 基础风控引擎。
- 模拟 OMS。
- 虚拟订单、成交、持仓。

验收标准：

- 策略只能生成交易意图。
- 交易意图必须经过风控。
- 模拟订单状态可追踪。
- 风控拒绝原因可审计。

### 阶段 5：最小实盘交易闭环

目标：在 miniQMT 权限、环境和 API 行为确认后，小心接入真实交易链路。

前置条件：

- miniQMT 运行环境确认。
- 账户查询、持仓查询、订单查询、成交查询能力确认。
- 下单、撤单 API 行为确认。
- 风控、OMS、日志、对账和人工暂停入口完成。

主要交付：

- `MiniQMTBrokerAdapter`。
- 账户、持仓、订单、成交查询。
- 下单、撤单。
- 对账任务。
- Web 监控和人工暂停。

验收标准：

- 所有真实订单都有策略来源、风控结果、OMS 状态和券商返回。
- 下单失败或未知状态不自动重试。
- 发现账户、订单或持仓不一致时暂停相关策略并提示人工确认。

### 阶段 6：多券商、多市场和高级能力

目标：在 A 股闭环稳定后扩展，而不是过早扩大复杂度。

可能交付：

- 盈立 OpenAPI 港股、美股适配器。
- 多市场交易日历。
- 多币种账户视图。
- 条件单、止盈止损、触发单、分批下单。
- 更完整的数据源和商业数据接入。
- LLM 研究助手。

验收标准：

- 新市场不破坏现有 A 股链路。
- 高级订单不绕过风控和 OMS。
- LLM 不进入真实下单链路。

## 6. 第一阶段推荐实施顺序

1. 建立 Python 后端和 PostgreSQL 基础工程。
2. 建立数据模型和迁移机制。
3. 实现 `DataProvider` 抽象。
4. 实现 `AkshareDataProvider` 和日线数据导入。
5. 建立数据质量检查。
6. 选择成熟回测库并做最小接入验证。
7. 实现一个最简单的 A 股日频回测任务。
8. 输出第一份 Markdown 研究报告。
9. 再决定是否补 React 控制台第一版。

这个顺序的理由是：先让数据和回测闭环跑通，再做界面。否则前端会很漂亮，但核心研究链路还没有真实反馈。

## 7. 关键依赖关系

- 没有标准化数据表，就不要让策略或回测直接接数据源。
- 没有数据质量检查，就不要把回测结论标记为可模拟盘。
- 没有 OMS 和风控状态机，就不要接真实下单。
- 没有对账和人工暂停，就不要做自动交易。
- 没有券商官方接口行为确认，就不要实现真实交易适配器。

## 8. 近期待决策事项

- 第一阶段先导入全 A 股，还是先导入 100 到 500 只样本股。
- PostgreSQL 本地部署方式：Docker 还是本机服务。
- 成熟回测库选型：Qlib、Backtrader、VectorBT、Zipline/Zipline Reloaded，或其他方案。
- 第一版研究任务入口：命令行、配置文件、API，还是 Web 表单。
- 第一版报告格式：先 Markdown，还是同时输出 HTML。
- React 控制台是在最小回测闭环后做，还是与后端并行做。
