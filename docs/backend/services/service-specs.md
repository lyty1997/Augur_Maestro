# M1 服务行为规范

版本：v0.1  
状态：草案，待用户确认  
最后更新：2026-05-15

## 0. 文档定位

本文档定义 M1 核心服务的行为边界。它比架构文档更贴近实现，但仍是设计文档，不是代码。

M1 服务分为五类：

- 数据网关服务。
- 股票池服务。
- 研究任务服务。
- 回测与归因服务。
- 报告服务。

所有服务通过 `rq_core` 的 DTO、Protocol 和 Service 接口协作。CLI 和 FastAPI 只能调用服务层，不能绕过服务直接读写数据库或外部 SDK。

## 1. 通用服务原则

### 1.1 依赖方向

```text
CLI / API
  -> Service
    -> Repository / Provider / Engine Adapter
      -> Database / External SDK / Filesystem
```

禁止：

- CLI 直接调用 AKShare。
- FastAPI Router 直接调用 SQLAlchemy session 修改核心表。
- 回测引擎直接读取数据库。
- `rq_core` 依赖外部 SDK。
- 策略、因子或报告代码直接访问 AKShare。

### 1.2 状态和错误

所有长任务必须有状态，不允许只有控制台输出。

状态必须表达：

- 是否开始。
- 是否运行中。
- 是否成功。
- 是否部分失败。
- 是否失败。
- 失败原因是什么。

错误不能只写在日志里，关键错误要进入数据库状态字段或事件表，方便报告和复盘引用。

### 1.3 幂等和重跑

M1 数据导入、质量检查、回测和报告可以由用户手动重跑。重跑策略：

- 数据导入使用唯一约束去重。
- 回测新跑一次生成新的 `backtest_id`。
- 报告新生成一次生成新的 `artifact_id`。
- 不覆盖历史回测结果，除非明确标记旧报告 `superseded`。

## 2. DataGatewayService

### 2.1 角色定位

DataGatewayService 负责把外部数据源转换成内部标准数据。它解决的问题是：上层研究和回测不能绑定 AKShare 这种临时数据源，否则未来切换到 miniQMT 或商业数据源会牵一发动全身。

### 2.2 输入

- Provider 名称，例如 `akshare`。
- 市场，例如 `CN_A`。
- 股票池 ID。
- 日期范围。
- 复权类型。
- 导入配置哈希。

### 2.3 输出

- `data_ingestion_runs` 批次。
- 标准化 `market_symbols`。
- 标准化 `trade_calendars`。
- 标准化 `daily_bars`。
- 失败标的和错误摘要。

### 2.4 状态机

```text
pending
  -> running
  -> succeeded
  -> partial_failed
  -> failed
  -> cancelled
```

状态规则：

- `pending`：批次已创建，未开始拉取。
- `running`：正在拉取、标准化或写入。
- `succeeded`：全部目标数据完成。
- `partial_failed`：部分标的或日期失败，但仍有数据成功入库。
- `failed`：任务无法产生可用数据。
- `cancelled`：用户或系统取消，M1 可先不实现主动取消。

### 2.5 关键规则

- Provider 失败要转换成统一错误。
- 原始数据字段变化要记录 provider 版本和错误上下文。
- 同一股票、日期、复权、来源不能重复入库。
- 不复权数据和前复权数据要可区分。
- 数据导入完成不等于可回测，必须经过质量检查。

## 3. DataQualityService

### 3.1 角色定位

DataQualityService 负责判断数据能不能进入研究结论。它不是为了追求完美数据，而是防止明显错误让回测失真。

### 3.2 输入

- 股票池 ID。
- 日期范围。
- 复权类型。
- 可选导入批次 ID。

### 3.3 输出

- `data_quality_runs`。
- `data_quality_issues`。
- 总体状态：`passed`、`warning`、`failed`。

### 3.4 检查项

M1 必须实现：

- 必填字段缺失。
- `high < low`。
- `open`、`close` 不在 `low` 到 `high` 区间。
- 成交量或成交额为负。
- 重复 `(symbol, trade_date, adjust, source)`。
- 交易日期不在交易日历。
- 同一标的在日期范围内大段缺失。
- 前复权序列出现明显断裂。
- 第二数据源关键字段抽样交叉验证。

M1 可暂缓：

- 与第二数据源全量逐字段对比。
- 复权因子独立校验。
- 停牌和涨跌停的完整交易规则还原。

### 3.5 阻断规则

- `error` 级问题默认阻断回测。
- `warning` 级问题允许回测，但报告必须列出。
- 如果用户显式允许 warning 数据，命令和报告都要记录该选择。

## 4. UniverseService

### 4.1 角色定位

UniverseService 负责研究股票池的候选、确认和正式成员管理。股票池是研究边界，不是交易白名单。

### 4.2 候选生成来源

- 人工 seed。
- AKShare 概念板块或行业分类。
- 规则过滤。
- AI 辅助解释或扩展。

AI 只能生成候选，不得自动批准。

### 4.3 候选状态机

```text
pending
  -> approved
  -> rejected
  -> ignored
```

规则：

- 只有 `approved` 可以进入正式研究股票池。
- `rejected` 表示明确不纳入。
- `ignored` 表示本轮暂不处理，不代表永远排除。
- 所有人工决策必须记录原因。

### 4.4 流动性和异常标的过滤

正式研究股票池需要排除明显不适合低频研究和未来实盘验证的高风险标的。

M1 过滤维度：

- ST 或风险警示。
- 上市时间过短。
- 长期停牌。
- 日均成交额过低。
- 换手率过低。
- 市值过小且波动异常。
- 成交量异常稀疏。
- 疑似容易被大资金操纵的小票股。

第一版采用保守阈值，并在配置文件中显式记录：

```yaml
universe_filters:
  exclude_st: true
  min_listed_days: 250
  min_avg_amount_20d: 50000000
  min_avg_turnover_rate_20d: 0.005
  max_suspended_days_250d: 20
  min_market_cap: 5000000000
  abnormal_volatility_filter: todo
```

### 4.5 候选理由规则

候选理由必须包含：

- 来源。
- 所属主题。
- 关联依据。
- 如果来自 AI，必须说明 AI 只是候选辅助。

禁止理由：

- “名称里有某个词”。
- “新闻里出现同名词”。
- “概念听起来相关”。
- 无法审计的主观判断。

## 5. DataCrossValidationService

### 4.6.1 角色定位

DataCrossValidationService 负责用第二数据源对 AKShare 主数据做关键字段校验。它解决的问题是：单一聚合数据源可能发生字段变化、复权异常或源站修正，导致回测结果看似可用但实际不稳。

### 4.6.2 输入

- 主数据源导入批次。
- 第二数据源 Provider。
- 股票池 ID。
- 日期范围。
- 抽样比例或抽样标的清单。
- 对比字段。

### 4.6.3 输出

- 差异摘要。
- 差异明细。
- 差异严重级别。
- 是否阻断回测。

### 4.6.4 M1 对比字段

M1 先覆盖：

- 交易日期是否一致。
- 收盘价是否差异超过阈值。
- 成交量是否差异超过阈值。
- 成交额是否差异超过阈值。
- 复权序列是否明显不一致。

### 4.6.5 阻断规则

- 单点轻微差异记录 warning。
- 大面积日期缺失、价格严重不一致、复权序列明显断裂可以标记 error。
- error 级差异默认阻断回测，除非用户显式将该任务标记为数据实验。

## 6. ResearchTaskService

### 5.1 角色定位

ResearchTaskService 负责把一次研究固定成可复现任务。它解决的问题是：以后看到一个报告时，必须知道当时用了哪些数据、股票池、参数和切分。

### 5.2 任务状态机

```text
created
  -> data_ready
  -> running
  -> backtested
  -> reported
  -> failed
  -> cancelled
```

M1 可先不实现 `cancelled` 的主动取消，但数据库状态要预留。

### 5.3 数据切分

默认策略：

1. 取研究配置中的 `start_date` 到 `end_date`。
2. 最近 1 年作为 `simulation`。
3. 剩余区间按时间顺序 8:2 切成 `modeling` 和 `backtest`。

规则：

- 不随机切分。
- 不使用晚于切分点的数据训练早期模型。
- 切分结果必须入库。
- 报告必须展示切分区间。

### 5.4 数据准备检查

进入回测前必须检查：

- 股票池存在且有 active 成员。
- 行情覆盖目标日期范围。
- 质量检查没有 error。
- 复权方式存在。
- 配置哈希已保存。

## 7. BacktestService

### 6.1 角色定位

BacktestService 负责组装标准化数据和回测引擎适配器。成熟回测库只负责计算，Augur_Maestro 负责数据版本、实验记录、风险假设和报告。

### 6.2 输入

- `task_id`。
- `engine`：`qlib` 或 `vectorbt`。
- `strategy_name`。
- 复权方式。
- 初始资金。
- 手续费。
- 滑点。
- 配置哈希。

### 6.3 输出

- `backtest_runs`。
- `backtest_metrics`。
- `backtest_equity_curve`。
- `backtest_orders`。
- `backtest_trades`。

### 6.4 规则

- 回测引擎不能直接访问 AKShare。
- 回测虚拟订单不能复用真实订单状态机。
- 回测结果失败时必须保留错误原因。
- 回测成本假设必须进入报告。
- 如果回测库对 A 股停牌、涨跌停或手续费支持不足，报告必须写明限制。

## 8. AttributionService

### 7.1 角色定位

AttributionService 负责把回测结果拆开看。它不负责美化策略，也不能把相关性包装成因果。

### 7.2 M1 归因维度

- 标的贡献。
- 主题贡献。
- 交易成本影响。
- 主要回撤片段。

后续再扩展：

- 因子贡献。
- 事件窗口贡献。
- 行业贡献。
- 市场状态贡献。

### 7.3 规则

- 每个归因项必须能追溯到回测数据。
- 无法归因时写“不足以归因”。
- 报告中使用“贡献”“相关”“拖累”等描述，避免写成确定因果。

## 9. ReportService

### 8.1 角色定位

ReportService 负责把研究任务、回测结果、归因和风险提示整理成 Markdown。

### 8.2 报告结构

M1 报告至少包含：

- 研究目标。
- 股票池和主题范围。
- 数据源和导入批次。
- 数据质量摘要。
- 时间范围和切分方式。
- 复权方式。
- 策略和回测参数。
- 回测指标。
- 权益曲线摘要。
- 归因分析。
- 失败和异常说明。
- 风险提示。
- 下一步建议。

### 8.3 报告安全

报告禁止包含：

- API Key。
- Secret。
- token。
- 真实账号。
- 身份证明。
- 真实资金隐私。
- 未脱敏的券商返回原文。

报告必须包含：

- “研究结论不等于实盘建议”。
- “回测不代表未来收益”。
- “数据源和股票池存在限制”。

## 10. 服务测试要求

M1 服务测试优先级：

1. 候选股票池必须人工确认。
2. 数据质量 error 必须阻断回测。
3. 数据切分不能乱序。
4. 回测不能直接访问 AKShare。
5. 报告必须包含风险提示。
6. 日志和报告不能泄漏敏感信息。

测试可以使用 fake provider 和小样本 fixture，不需要每次测试真实访问 AKShare。
