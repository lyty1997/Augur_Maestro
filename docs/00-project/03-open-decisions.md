# RobustQuant 待决策清单

版本：v0.1  
状态：部分确认

这份文档记录后续需要逐步确认的问题。它的作用是防止我们在架构还不清楚时过早写死实现。

## 1. 市场和券商接入优先级

已确认：

- M1 先做 A 股研究回测闭环。
- M1 不接真实交易接口，只保留统一券商适配器抽象和占位实现。
- 盈立 OpenAPI 申请结果不确定，港股、美股和盈立正式接入不作为 M1 研究闭环依赖。
- 盈立反馈申请 OpenAPI 需要提供登录、下单、改单、撤单源码截图；该事项属于券商申请材料，不作为 RobustQuant 开发交付。
- miniQMT 和盈立 OpenAPI 的只读查询测试不构成真实交易；任何会向券商提交委托、撤单、条件单、预埋单或改变券商侧订单状态的调用，都按真实交易处理。
- 不允许用真实下单、真实撤单或真实条件单接口做连通性探测、冒烟测试或权限测试。
- 真实交易能力默认关闭，必须经过实盘开关、人工确认、交易时间检查、风控、OMS、账户/标的白名单和对账设计后才能启用。

待确认：

- 东北证券 miniQMT 是否已具备交易权限和 API 权限。
- 第一条实盘闭环后续是否优先接 miniQMT；需等待 miniQMT 权限、运行环境和接口能力确认。
- 盈立 OpenAPI 若后续申请通过，再根据官方 API 手册确认港股、美股行情和交易权限。
- 盈立是否提供 sandbox；如果 PDF 未说明，按 `unknown_by_pdf` 处理，不运行真实下单、改单、撤单。

建议：

- A 股研究闭环先只依赖数据网关和回测引擎，不依赖真实下单能力。
- 券商 API 先保留适配器抽象，不急着实现具体交易接口。
- 盈立官方 PDF 解析按 [17-yingli-openapi-reference.md](../30-trading/17-yingli-openapi-reference.md) 执行；源码截图材料不进入开发任务。
- `TradingGateway` 统一券商交易网关设计见 [18-broker-trading-gateway-design.md](../30-trading/18-broker-trading-gateway-design.md)；正式实现前先确认能力模式、只读联调边界和盈立 `unknown_by_pdf` 项。
- 等“查询账户、查询持仓、下单、撤单、查询成交、对账、日志、告警”跑通后，再扩展第二个券商。

## 2. miniQMT 运行环境

已确认：

- miniQMT 支持无人值守运行。

待确认：

- miniQMT 是否必须运行在 Windows。
- 是否必须保持客户端登录。
- 是否支持自动重连。
- 行情和交易接口是否在同一个客户端中提供。

可能影响：

- 如果 miniQMT 必须依赖 Windows 桌面客户端，系统需要设计一个 Windows 执行节点。
- 如果主服务部署在 Linux 云服务器，Windows 执行节点和主服务之间还需要一层内部通信。

## 3. 数据源选择

已确认：

- 生产级目标优先使用券商接口作为 A 股行情和账户相关数据源。
- 当前还没有可用券商数据接口，M1 默认使用 AKShare 作为 A 股历史行情研究启动源。
- 策略、回测和研究 Agent 只能读取内部标准化数据表，不能直接依赖某个外部数据源 SDK。
- 数据接入必须通过 `DataProvider` 抽象，并写入 PostgreSQL 标准表。
- M1 只导入用户熟悉主题相关标的，不先导入全 A 股。主题包括算力、电力、半导体、人工智能、芯片、电池、通信、银行、CPO、机器人、存储、有色金属、影视院线。
- M1 需要加入第二数据源用于交叉验证，第一版选择 BaoStock；Tushare Pro 或本地 CSV/Parquet 作为后续备用方案。

待确认：

- 财务数据来自哪里。
- 公告、新闻、研报等文本数据来自哪里。
- BaoStock 字段、限流或维护状态不满足 M1 需要时，是否切换到本地 CSV/Parquet 作为备用校验源。
- 是否需要购买商业数据源，以及商业数据源的授权范围是否允许用于个人量化研究。
- 主题股票池第一版具体标的清单。

建议：

- M1 先用最少数据跑通研究回测闭环：A 股日线行情、交易日历、复权因子、停牌状态、基础股票列表。
- 数据源路线建议分三步：
  1. 先定义内部标准化数据模型和 `DataProvider` 抽象。
  2. 研究期接入 AKShare，优先保证可复现和可替换。
  3. miniQMT 或其他券商行情接口可用后，新增券商数据 Provider，并用同一套数据质量检查对比临时数据源和券商数据。
- 不急着做复杂因子库，先确保交易日历、复权、停牌、手续费、滑点处理正确。
- 具体数据源专项路线见 [06-data-source-roadmap.md](../10-research-data/06-data-source-roadmap.md)，项目总体路线见 [05-technical-roadmap.md](05-technical-roadmap.md)。

## 4. 策略接口设计

已确认：

- 策略输出使用代码 + 名称的用户可见表达法，例如 `688981.SH 中芯国际`；内部仍以标准代码作为主键。
- 数据默认导入最近 10 年；最近 1 年作为模拟数据，此前数据按时间顺序 8:2 切分为建模数据和回测数据。
- 切分必须按时间顺序，不能随机打乱，避免未来函数。
- 实盘阶段第一批优先选择价格较低、流动性好的宽基 ETF 验证链路；个股实盘后置。模拟盘可以更自由地覆盖个股和主题股票池。

待确认：

- 策略按日频、分钟级，还是事件驱动运行。
- 策略输出“买卖数量”，还是输出“目标仓位”。
- 同一账户内多个策略如何分配资金。
- 多策略同时想买卖同一只股票时如何合并订单。

建议：

- M1 优先使用目标仓位模型。例如策略说“让腾讯控股变成账户 10% 仓位”，系统自己计算需要买卖多少。
- 目标仓位模型更容易做组合管理和风控。

## 5. 风控参数

已确认：

- 非交易时间不得向券商提交真实下单或挂单请求。
- 非交易时间触发的条件单、策略信号或消息面信号，只能进入待审计交易意图或待处理状态；到下一个有效交易窗口后必须重新经过行情读取、风控检查和 OMS 状态检查。

待确认：

- 单笔订单最大金额。
- 单只股票最大持仓比例。
- 单日最大交易次数。
- 单日最大亏损阈值。
- 是否允许市价单。
- 是否允许盘前盘后美股交易。
- 是否需要强制只用限价单。

建议：

- 初期默认保守，只允许限价单。
- 初期资金规模小一点，用真实链路验证系统稳定性。

## 6. 技术栈

已确认：

- 后端使用 FastAPI。
- 数据库使用 PostgreSQL。
- 前端使用 React。
- PostgreSQL M1 本地部署，优先 Docker Compose，本机 PostgreSQL 服务作为备选。
- Python 项目管理使用 `venv` + `requirements.txt` 起步。
- 第一版任务入口使用 YAML 配置文件 + CLI 命令。
- CLI 框架使用 Typer。
- CLI 必须提供完整 `--help`，README 必须写清常用命令、配置文件示例和运行顺序。
- 数据库访问使用 SQLAlchemy 2.x，数据库迁移使用 Alembic。
- 配置文件分为 `.env` 和 `configs/*.yaml`；`.env` 不进入 Git，`configs/*.yaml` 可以进入 Git。
- 中间产物和输出目录不进入代码库。
- 代码架构采用拆分 kernel 的方式，尽可能保证高内聚、低耦合。
- M1 数据模型按 [08-m1-data-model.md](../20-m1/08-m1-data-model.md) 执行，真实交易相关模型暂不进入 M1。
- M1 Kernel 接口类按 [09-m1-kernel-interfaces.md](../20-m1/09-m1-kernel-interfaces.md) 执行，核心 DTO 和 Protocol 放在 `rq_core`，不依赖 FastAPI、Typer、SQLAlchemy、AKShare、Qlib 或 VectorBT。
- M1 不引入 Redis、Celery、Dramatiq 等后台任务队列。

待确认：

- 后续何时引入 Redis。
- 后续后台任务使用 APScheduler、Celery 还是 Dramatiq。

建议：

- M1 可以先用 FastAPI + PostgreSQL + React 搭出最小研究回测闭环。
- Redis 和后台任务框架等到出现明确的异步任务、队列或缓存需求后再定，避免过早引入复杂度。
- M1 的数据导入、质量检查和最小回测优先用 CLI 同步执行，方便调试和复现。

## 8. AI/LLM 的里程碑边界

已确认：

- M1 可以前置 AI 辅助选股，但只允许生成候选股票池，不允许自动进入实盘交易链路。
- AI 候选股票池必须记录标的、名称、主题、来源、候选理由和置信度，并经过用户确认后才能进入正式研究股票池。
- AI 或消息面不能仅凭关键词、同名或概念碰瓷选股交易；例如“美国总统特朗普抵达天坛”不能据此选择带有“天坛”字样的股票建仓。
- M1 允许 AI 扩展候选清单，但 AI 扩展结果仍然只是候选，必须人工确认，并且必须记录来源、理由和置信度。

待确认：

- LLM 在研究回测 Agent 中负责哪些环节：解释结果、生成候选因子、生成策略草案、生成报告，还是全部参与。
- LLM 生成代码是否允许自动运行。

建议：

- LLM 可以作为研究助手参与研究回测 Agent，但不进入实盘交易链路。
- LLM 生成的代码只能在沙箱里运行，并且需要人工审核。

## 9. 研究回测 Agent 设计

已确认：

- Agent 在 M1 只做 A 股研究回测闭环。
- 回测引擎优先选择成熟库，不自研完整回测框架。
- 交易接口暂时只做抽象占位，不阻塞研究 Agent。
- 回测库先并行最小验证 Qlib 和 VectorBT，再选择最适合 RobustQuant 的主执行器。
- 第一版报告格式使用 Markdown。
- 研究报告必须包含基础归因分析，回答主要收益和亏损来自哪些标的、主题、因子、时间窗口、事件窗口和交易成本。
- 第一版主题股票池不要求用户逐只手工输入，可以由 AKShare 概念板块、行业分类、公开资料和 AI 辅助生成候选，再由用户确认。
- 研究、数据、股票池、回测和报告之间通过 M1 kernel 接口协作；CLI 和 FastAPI 只能调用服务层，不能绕过 kernel 直接写数据库或调用外部 SDK。

待确认：

- Agent 的研究任务入口是什么：自然语言目标、配置文件、Web 表单，还是三者都支持。
- 自动选股使用哪些基础规则，例如流动性、上市时间、停牌、ST、成交额、市值区间。
- 因子挖掘先支持哪些因子类型：技术因子、量价因子、基本面因子、事件因子。
- 归因分析第一版的维度权重和展示方式：先做标的/主题/成本/回撤，还是同步纳入因子和事件窗口。
- 分时段训练采用哪种切分方式：固定训练/验证/测试，还是滚动 walk-forward。
- Qlib 和 VectorBT 并行最小验证后，M1 主回测执行器选型结论和分工。
- 模拟盘是否先离线模拟，还是接入实时行情做 paper trading。
- 后续报告是否增加 HTML/PDF。

建议：

- M1 先做规则驱动的 Agent 编排，不急着让 LLM 自动生成可执行代码。
- 回测库只能作为执行引擎，RobustQuant 仍需保留自己的数据模型、实验记录、风控假设和报告格式。
- 先把数据版本、实验参数、回测结果、报告产物记录完整，避免研究结果不可复现。
- Agent 输出策略草案后必须进入人工审核状态，不能直接进入实盘。

## 10. 消息面与宏观事件信号

已确认：

- 消息面注入不能仅凭关键词匹配选股或交易，必须经过实体识别、标的关联、行业/基本面相关性和可审计理由校验。
- 没有真实关联证据的文本只能进入新闻或事件观察库，不能形成交易意图。

待确认：

- 消息面数据源选择：财经新闻源、交易所公告、券商数据、第三方宏观日历，还是多源融合。
- M1 宏观事件覆盖哪些：CPI、PPI、非农就业、失业率、ADP、初请失业金、FOMC、PMI、GDP。
- 是否需要美股盘前/盘中快速事件分析，还是先做日频事件研究。
- 事件数据是否能提供准确的预期值、公布值、修正值和发布时间。
- 事件发布后分析哪些资产反应：指数、行业 ETF、个股、美债收益率、美元指数、黄金、原油。
- 事件信号先用于报告和风控提示，还是 M1 就注入策略回测。
- LLM 是否参与新闻摘要和事件解释；如果参与，结构化结果如何校验。
- 10 分钟快速注入 SLA 的起点如何定义：事件官方发布时间、数据源到达时间、系统接收时间、行情确认时间，还是全部记录。
- 快速风险退出 M1 覆盖哪些资产：美股科技、半导体、存储芯片、纳斯达克指数 ETF，还是同步覆盖 A 股相关行业。
- 快速风险退出允许的最大动作幅度：只暂停开仓、减仓 30%、减仓 50%、还是允许清仓。
- 快速事件链路失效时默认动作是什么：只告警、暂停开仓、还是触发人工确认。

建议：

- 先建立事件日历和事件结果数据模型，不急着做实时交易响应。
- M1 优先做事件窗口研究：发布前后收益、波动、成交量、回撤、相关资产联动。
- 事件信号默认先作为因子、过滤器或风控上下文，不能直接变成实盘交易指令。
- 对 CPI、非农这类高波动事件，默认设计“提高风控严格度”而不是“自动抢方向”。
- 对重大利空事件，允许设计防守型快速风险退出：暂停开仓、降低相关行业暴露、清理指定风险资产。
- 快速风险退出链路目标：重要事件发布或行情确认后，10 分钟内让策略运行时和风控读到结构化事件上下文。
- 必须记录全链路时间戳：事件发布时间、系统接收时间、分析完成时间、信号写入时间、策略读取时间、交易意图生成时间。
- 快速风险退出只能降低风险暴露，不能用于自动加仓、追涨或绕过风控。

## 11. 条件单和高级订单能力

已确认：

- 条件单、止盈止损、触发单、分批下单等能力需要在后续架构中预留，但不进入 M1 研究回测闭环的实现范围。

待确认：

- 条件单由本地 OMS 托管，还是使用券商原生条件单能力。
- 触发条件先支持哪些类型：价格触发、时间触发、涨跌幅触发、回撤触发、事件触发。
- 条件单触发后的订单类型、价格保护、有效期和撤单规则。
- 条件单状态机、审计日志、人工暂停和异常恢复规则。

建议：

- 后续设计条件单时，必须先确认券商 API 是否原生支持；如果本地托管，需要额外设计进程存活、行情延迟、断线恢复和重复触发防护。
- 条件单不能绕过风控和 OMS，触发后仍然只能生成交易意图或订单请求，必须经过统一风控检查。

## 12. 本轮详细设计后的优先讨论问题

本节来自 2026-05-15 参考 Narrative-Maestro 文档颗粒度后补充的专项设计。以下问题建议在进入 M1 编码前确认或排序。

### 12.1 M1 是否做 React 只读控制台

已确认：

- M1 主入口是 CLI。
- React 是长期技术栈方向。
- M1 前端如实现，只能只读展示研究状态，不能提供真实交易入口。
- M1 当前优先先跑通 CLI 研究闭环，React 只读控制台不进入第一条关键路径。

待确认：

- CLI 跑通后是否把 React 只读控制台作为 M1 收口增强，还是放到 M2。

建议：

- 先完成 CLI 研究闭环。
- 若时间允许，再补只读控制台查看导入批次、质量检查、研究任务、回测和报告。

### 12.2 Qlib 和 VectorBT 的分工

已确认：

- M1 先并行最小验证 Qlib 和 VectorBT。
- 至少要跑通一个 A 股日频最小回测。
- 验证后选择最适合的主执行器，选择依据包括 A 股适配成本、数据接入复杂度、维护活跃度、可解释性、性能和报告集成成本。

待确认：

- 并行验证后的主执行器选择。
- 未被选为主执行器的一方是否保留为对照验证工具。

建议：

- 先用同一份标准化数据各跑一个最小策略。
- 用 A 股适配成本、可解释性、维护活跃度和报告集成成本决定主执行器。

### 12.3 AI 候选股票池边界

已确认：

- AI 只能生成候选，不得自动进入正式股票池或实盘链路。
- 候选必须人工确认。
- 候选理由不能只靠关键词、同名或概念碰瓷。
- M1 允许 AI 根据公开资料扩展候选清单，但必须记录来源、理由和置信度。

待确认：

- 如果允许扩展，引用来源和置信度如何要求。

建议：

- 第一版先让 AI 做解释、归类和补充理由。
- 候选扩展可以保留，但必须导出人工确认清单。

### 12.4 数据源交叉验证是否进入 M1

已确认：

- AKShare 是研究启动源，不是最终生产真相源。
- 后续应引入券商数据或其他数据源校验。
- M1 需要加入第二数据源做交叉验证。
- 第二数据源第一版选择 BaoStock。
- 交叉验证第一版先覆盖关键字段抽样，不做全量逐字段对比；关键字段至少包括收盘价、成交量、成交额、交易日和复权一致性。

待确认：

- BaoStock 如果字段缺失、限流或维护状态不满足 M1 需要，是否切换到本地 CSV/Parquet 作为备用校验源。
- 关键字段抽样比例和差异阈值。

建议：

- 第二数据源先做关键字段抽样交叉验证，不阻塞主数据导入，但会在报告中标记数据差异风险。

### 12.5 第一版主题股票池标的清单

已确认：

- M1 不导入全 A 股。
- 主题范围已确认。
- 股票池需要排除表现异常、容易被大资金操纵的小票股。
- 换手率低、成交额低、长期停牌、流动性差的标的不进入正式研究池。
- 第一版采用保守过滤阈值：最小上市天数 250、20 日均成交额不低于 5000 万元、20 日均换手率不低于 0.5%、近 250 个交易日停牌天数不超过 20、总市值不低于 50 亿元。

待确认：

- 每个主题第一批纳入哪些标的。
- 是否限定每个主题最多候选数量。
- 异常波动过滤口径。

建议：

- 每个主题先控制在少量用户熟悉标的。
- 先验证工程闭环，再扩大全市场研究范围。

### 12.6 实盘前风控阈值

已确认：

- M1 不接真实交易。
- 后续实盘前必须有风控、OMS、交易时间、白名单、人工确认和对账。
- 实盘阶段先用比较便宜、流动性好的宽基 ETF 小范围验证策略链路，个股实盘后置。
- 模拟盘阶段可以更自由地覆盖个股、行业 ETF 和主题股票池。

待确认：

- 第一版真实交易是否只允许限价单。
- 单笔订单最大金额。
- 单只股票最大持仓比例。
- 单日最大交易次数。
- 单日最大亏损阈值。
- 快速风险退出最大允许减仓比例。
- 第一批宽基 ETF 白名单。

### 12.7 盈立 OpenAPI 官方 PDF 解析

已确认：

- 盈立反馈 OpenAPI 申请需要登录、下单、改单、撤单源码截图。
- 源码截图属于券商申请材料事项，不作为 RobustQuant 开发交付。
- 不允许用真实下单、改单、撤单接口做连通性测试或截图演示。
- 已拿到盈立繁体中文 PDF 官方文档，位置为 `券商API/盈立/API文档/`。
- 当前已知 OpenAPI 主要覆盖直接管理交易、查看账户信息、查询行情变化和接收订单/持仓/报价实时变动。
- 交易相关 PDF 包括 `交易開放API接口文檔V1.0-20201029(繁).pdf`，行情相关 PDF 包括 `基礎報價開放API(繁)_20201029.pdf` 和 `報價推送(繁)_20201029.pdf`。
- 交易 PDF 初步摘录显示普通下单 endpoint 为 `/stock-order-server/open-api/entrust-order`，改单/撤单共用 `/stock-order-server/open-api/modify-order`。
- 交易 PDF 初步摘录显示 `modify-order` 通过 `actionType=1` 表示改单、`actionType=0` 表示撤单；最终语义仍需官方或逐页 PDF 校对确认。
- 交易 PDF 初步摘录显示下单响应 `data.entrustId` 可用于查询订单、修改订单和取消订单；是否可稳定作为内部 `broker_order_id` 仍需确认。

待确认：

- 盈立是否提供 sandbox。
- 改单是原生 modify API，还是 cancel + replace 流程。
- 交易 API base URL。
- `X-Request-Id` 长度、有效期和重复请求语义；当前 PDF 摘录存在 19 位和 30 位两种描述。
- 下单 body `serialNo` 与 header `X-Request-Id` 的关系。
- `X-Sign` 签名原文：交易 API 当前摘录以 body 为主，基础报价 API 摘录为 header+body 有序拼接，需确认能否分策略处理。
- `X-Type` 是否必填及枚举。
- 登录手机号、登录密码、交易密码使用的 RSA 公钥和字段名。
- `data.entrustId` 在下单、改单、撤单响应中分别代表原委托 ID、新委托 ID 还是申请编号。
- 订单类型、价格类型、`entrustProp`、`exchangeType`、`sessionType`、盘前盘后/暗盘规则。
- 错误码、订单状态码、`finalStateFlag` 和状态流转完整枚举。
- token 有效期、刷新方式、多会话冲突语义、HTTP 频率限制、WebSocket 订阅上限和 IP 白名单生效规则。

建议：

- 只解析 PDF，为未来正式接入做设计依据。
- PDF 中找不到的 sandbox 和改单语义，标记为 `unknown_by_pdf`；不把券商申请截图格式作为当前开发阻塞项。
- 具体调用栈、字段映射、只读查询 DTO 和签名策略按 [19-usmart-openapi-call-design.md](../30-trading/19-usmart-openapi-call-design.md) 继续维护；统一网关能力模式和安全边界按 [18-broker-trading-gateway-design.md](../30-trading/18-broker-trading-gateway-design.md) 维护。
